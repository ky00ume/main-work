Claude가 효과적으로 발견하고 사용할 수 있는 스킬(Skill)을 작성하는 방법을 알아봅니다.

일관된 명명 패턴을 사용하면 스킬을 더 쉽게 참조하고 논의할 수 있습니다. 스킬 이름에는 동명사 형태(동사 + -ing)를 사용하는 것을 고려하세요. 이는 스킬이 제공하는 활동이나 기능을 명확하게 설명합니다.



name 필드에는 소문자, 숫자, 하이픈만 사용해야 합니다.



좋은 명명 예시 (동명사 형태):

processing-pdfs, analyzing-spreadsheets, managing-databases, testing-code, writing-documentation



허용 가능한 대안:

명사구: pdf-processing, spreadsheet-analysis

행동 지향: process-pdfs, analyze-spreadsheets



피해야 할 것:

모호한 이름: helper, utils, tools

지나치게 포괄적: documents, data, files

예약어: anthropic-helper, claude-tools

스킬 컬렉션 내에서 일관성 없는 패턴



일관된 명명은 문서와 대화에서 스킬을 쉽게 참조하고, 한눈에 스킬의 기능을 이해하며, 여러 스킬을 정리하고 검색하고, 전문적이고 일관된 스킬 라이브러리를 유지하는 데 도움이 됩니다.

깊이 중첩된 참조를 피하라


Claude는 다른 참조 파일에서 참조되는 파일을 부분적으로만 읽을 수 있습니다. 중첩된 참조를 만나면 Claude는 전체 파일을 읽는 대신 head -100과 같은 명령으로 콘텐츠를 미리보기할 수 있어, 불완전한 정보를 얻을 수 있습니다.



SKILL.md에서 한 단계 깊이로 참조를 유지하세요. 모든 참조 파일은 SKILL.md에서 직접 링크되어야 Claude가 필요할 때 파일 전체를 읽을 수 있습니다.



원문:

Bad example: Too deep:

# SKILL.md
See [advanced.md](advanced.md)...

# advanced.md
See [details.md](details.md)...

# details.md
Here's the actual information...
Good example: One level deep:

# SKILL.md

**Basic usage**: [instructions in SKILL.md]
**Advanced features**: See [advanced.md](advanced.md)
**API reference**: See [reference.md](reference.md)
**Examples**: See [examples.md](examples.md)
번역:

나쁜 예시: 너무 깊음: SKILL.md → advanced.md → details.md 순으로 중첩 참조하면 실제 정보에 도달하기까지 너무 많은 단계가 필요합니다.

좋은 예시: 한 단계 깊이: SKILL.md에서 advanced.md, reference.md, examples.md를 모두 직접 참조합니다.

## Research synthesis workflow

Copy this checklist and track your progress:

```
Research Progress:
- [ ] Step 1: Read all source documents
- [ ] Step 2: Identify key themes
- [ ] Step 3: Cross-reference claims
- [ ] Step 4: Create structured summary
- [ ] Step 5: Verify citations
```

**Step 1: Read all source documents**

Review each document in the `sources/` directory. Note the main arguments and supporting evidence.

**Step 2: Identify key themes**

Look for patterns across sources. What themes appear repeatedly? Where do sources agree or disagree?

**Step 3: Cross-reference claims**

For each major claim, verify it appears in the source material. Note which source supports each point.

**Step 4: Create structured summary**

Organize findings by theme. Include:
- Main claim
- Supporting evidence from sources
- Conflicting viewpoints (if any)

**Step 5: Verify citations**

Check that every claim references the correct source document. If citations are incomplete, return to Step 3.
번역:

## 리서치 종합 워크플로우

이 체크리스트를 복사하여 진행 상황을 추적하세요:

```
리서치 진행 상황:
- [ ] 1단계: 모든 소스 문서 읽기
- [ ] 2단계: 핵심 주제 파악
- [ ] 3단계: 주장 교차 검증
- [ ] 4단계: 구조화된 요약 작성
- [ ] 5단계: 인용 검증
```

**1단계: 모든 소스 문서 읽기**

`sources/` 디렉터리의 각 문서를 검토합니다. 주요 주장과 뒷받침하는 근거를 메모하세요.

**2단계: 핵심 주제 파악**

소스 전반에서 패턴을 찾으세요. 반복적으로 나타나는 주제가 무엇인가? 소스가 동의하거나 불일치하는 부분은 어디인가?

**3단계: 주장 교차 검증**

각 주요 주장에 대해 소스 자료에 나타나는지 확인합니다. 각 요점을 뒷받침하는 소스를 메모하세요.

**4단계: 구조화된 요약 작성**

주제별로 발견 사항을 정리합니다. 포함할 내용:
- 주요 주장
- 소스의 뒷받침 증거
- 상충되는 관점 (있는 경우)

**5단계: 인용 검증**

모든 주장이 올바른 소스 문서를 참조하는지 확인합니다. 인용이 불완전하면 3단계로 돌아가세요.


이 예시는 코드가 필요 없는 분석 작업에 워크플로우가 어떻게 적용되는지 보여줍니다. 체크리스트 패턴은 복잡한 다단계 프로세스에 모두 적용됩니다.

원문:

## PDF form filling workflow

Copy this checklist and check off items as you complete them:

```
Task Progress:
- [ ] Step 1: Analyze the form (run analyze_form.py)
- [ ] Step 2: Create field mapping (edit fields.json)
- [ ] Step 3: Validate mapping (run validate_fields.py)
- [ ] Step 4: Fill the form (run fill_form.py)
- [ ] Step 5: Verify output (run verify_output.py)
```

**Step 1: Analyze the form**
Run: `python scripts/analyze_form.py input.pdf`
This extracts form fields and their locations, saving to `fields.json`.

**Step 2: Create field mapping**
Edit `fields.json` to add values for each field.

**Step 3: Validate mapping**
Run: `python scripts/validate_fields.py fields.json`
Fix any validation errors before continuing.

**Step 4: Fill the form**
Run: `python scripts/fill_form.py input.pdf fields.json output.pdf`

**Step 5: Verify output**
Run: `python scripts/verify_output.py output.pdf`
If verification fails, return to Step 2.

## PDF 양식 채우기 워크플로우

이 체크리스트를 복사하여 완료된 항목을 체크하세요:

```
작업 진행 상황:
- [ ] 1단계: 양식 분석 (analyze_form.py 실행)
- [ ] 2단계: 필드 매핑 생성 (fields.json 편집)
- [ ] 3단계: 매핑 검증 (validate_fields.py 실행)
- [ ] 4단계: 양식 채우기 (fill_form.py 실행)
- [ ] 5단계: 출력 확인 (verify_output.py 실행)
```

**1단계: 양식 분석**
실행: `python scripts/analyze_form.py input.pdf`
양식 필드와 위치를 추출하여 `fields.json`에 저장합니다.

**2단계: 필드 매핑 생성**
`fields.json`을 편집하여 각 필드에 값을 추가합니다.

**3단계: 매핑 검증**
실행: `python scripts/validate_fields.py fields.json`
계속하기 전에 검증 오류를 수정하세요.

**4단계: 양식 채우기**
실행: `python scripts/fill_form.py input.pdf fields.json output.pdf`

**5단계: 출력 확인**
실행: `python scripts/verify_output.py output.pdf`
검증에 실패하면 2단계로 돌아가세요.

## Commit message format

Generate commit messages following these examples:

**Example 1:**
Input: Added user authentication with JWT tokens
Output:
```
feat(auth): implement JWT-based authentication

Add login endpoint and token validation middleware
```

**Example 2:**
Input: Fixed bug where dates displayed incorrectly in reports
Output:
```
fix(reports): correct date formatting in timezone conversion

Use UTC timestamps consistently across report generation
```

**Example 3:**
Input: Updated dependencies and refactored error handling
Output:
```
chore: update dependencies and refactor error handling

- Upgrade lodash to 4.17.21
- Standardize error response format across endpoints
```

Follow this style: type(scope): brief description, then detailed explanation.
번역:

이 예시들을 따라 커밋 메시지를 생성하세요. 형식: type(scope): 간단한 설명, 그 다음 상세한 설명. 예시 1은 JWT 기반 인증 추가를 feat(auth)로, 예시 2는 보고서의 날짜 표시 버그 수정을 fix(reports)로, 예시 3은 의존성 업데이트와 오류 처리 리팩터링을 chore로 분류합니다.

평가와 반복


평가를 먼저 만들어라


광범위한 문서를 작성하기 전에 평가(Evaluation)를 만드세요. 이렇게 하면 스킬이 상상의 문제가 아닌 실제 문제를 해결하도록 보장합니다.



평가 주도 개발(Evaluation-driven Development):

1. 격차 파악: 스킬 없이 대표적인 작업에 Claude를 실행합니다. 구체적인 실패나 누락된 컨텍스트를 문서화합니다.

2. 평가 생성: 이러한 격차를 테스트하는 세 가지 시나리오를 만듭니다.

3. 기준선 설정: 스킬 없이 Claude의 성능을 측정합니다.

4. 최소한의 지시 작성: 격차를 해결하고 평가를 통과하기에 충분한 콘텐츠만 작성합니다.

5. 반복: 평가를 실행하고, 기준선과 비교하고, 개선합니다.



이 접근 방식은 결코 현실화되지 않을 수 있는 요구사항을 예측하는 대신 실제 문제를 해결하도록 보장합니다.



평가 구조:

{
  "skills": ["pdf-processing"],
  "query": "Extract all text from this PDF file and save it to output.txt",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "Successfully reads the PDF file using an appropriate PDF processing library or command-line tool",
    "Extracts text content from all pages in the document without missing any pages",
    "Saves the extracted text to a file named output.txt in a clear, readable format"
  ]
}


이 예시는 간단한 테스트 루브릭을 가진 데이터 기반 평가를 보여줍니다. 현재 이러한 평가를 실행하는 내장 방법은 없습니다. 사용자가 자체 평가 시스템을 만들 수 있습니다. 평가는 스킬 효과를 측정하기 위한 진실의 원천입니다.



Claude와 함께 스킬을 반복적으로 개발하라


가장 효과적인 스킬 개발 프로세스는 Claude 자체를 활용합니다. 하나의 Claude 인스턴스("Claude A")와 작업하여 다른 인스턴스("Claude B")가 사용하는 스킬을 만듭니다. Claude A는 지시사항을 설계하고 개선하는 데 도움을 주고, Claude B는 실제 작업에서 테스트합니다. 이것이 가능한 이유는 Claude 모델이 효과적인 에이전트 지시사항을 작성하는 방법과 에이전트가 필요로 하는 정보를 모두 이해하기 때문입니다.



새 스킬 만들기:



1. 스킬 없이 작업을 완료합니다: 일반 프롬프팅으로 Claude A와 문제를 해결합니다. 작업하면서 자연스럽게 컨텍스트를 제공하고, 선호사항을 설명하고, 절차적 지식을 공유하게 됩니다. 반복적으로 제공하는 정보가 무엇인지 주목하세요.



2. 재사용 가능한 패턴을 파악합니다: 작업 완료 후, 유사한 향후 작업에 유용할 컨텍스트를 파악합니다.



예: BigQuery 분석을 수행했다면 테이블 이름, 필드 정의, 필터링 규칙(예: "항상 테스트 계정 제외"), 일반적인 쿼리 패턴을 제공했을 수 있습니다.



3. Claude A에게 스킬 생성을 요청합니다:

원문:

"Create a Skill that captures this BigQuery analysis pattern we just used. Include the table schemas, naming conventions, and the rule about filtering test accounts."

번역:

"방금 사용한 BigQuery 분석 패턴을 캡처하는 스킬을 만들어줘. 테이블 스키마, 명명 규칙, 테스트 계정 필터링 규칙을 포함해줘."



Claude 모델은 스킬 형식과 구조를 기본적으로 이해합니다. Claude가 스킬을 만들도록 하기 위해 특별한 시스템 프롬프트나 "스킬 작성" 스킬이 필요하지 않습니다. 스킬을 만들어 달라고 요청하면 적절한 프론트매터와 본문 콘텐츠가 있는 올바르게 구조화된 SKILL.md 콘텐츠를 생성합니다.



4. 간결함을 검토합니다:

원문:

"Remove the explanation about what win rate means - Claude already knows that."

번역:

"승률이 뭔지 설명하는 부분 제거해줘 — Claude는 이미 알고 있어."



5. 정보 아키텍처를 개선합니다:

원문:

"Organize this so the table schema is in a separate reference file. We might add more tables later."

번역:

"테이블 스키마를 별도의 참조 파일로 분리해서 정리해줘. 나중에 테이블을 더 추가할 수도 있어."



6. 유사한 작업에서 테스트합니다: 스킬이 로드된 Claude B(새 인스턴스)를 사용하여 관련 유스 케이스에서 테스트합니다. Claude B가 올바른 정보를 찾는지, 규칙을 올바르게 적용하는지, 작업을 성공적으로 처리하는지 관찰합니다.



7. 관찰에 기반하여 반복합니다:

원문:

"When Claude used this Skill, it forgot to filter by date for Q4. Should we add a section about date filtering patterns?"

번역:

"Claude가 이 스킬을 사용할 때 Q4 날짜 필터링을 잊었어. 날짜 필터링 패턴에 대한 섹션을 추가해야 할까?"



기존 스킬 반복 개선:



동일한 계층적 패턴이 스킬 개선 시에도 계속됩니다. 다음을 번갈아 수행합니다:

Claude A와 작업 (스킬을 개선하는 데 도움을 주는 전문가) / Claude B로 테스트 (스킬을 사용하여 실제 작업을 수행하는 에이전트) / Claude B의 동작을 관찰하고 인사이트를 Claude A에게 전달



1. 실제 워크플로우에서 스킬을 사용합니다: Claude B(스킬이 로드된)에게 테스트 시나리오가 아닌 실제 작업을 부여합니다.



2. Claude B의 동작을 관찰합니다: 어디서 어려움을 겪는지, 성공하는지, 예상치 못한 선택을 하는지 주목합니다.



관찰 예시: "Claude B에게 지역별 영업 보고서를 요청했을 때, 쿼리를 작성했지만 스킬에서 이 규칙을 언급하고 있음에도 테스트 계정 필터링을 잊었다."



3. Claude A에게 돌아가 개선합니다: 현재 SKILL.md를 공유하고 관찰한 내용을 설명합니다.

원문:

"I noticed Claude B forgot to filter test accounts when I asked for a regional report. The Skill mentions filtering, but maybe it's not prominent enough?"

번역:

"지역 보고서를 요청했을 때 Claude B가 테스트 계정 필터링을 잊는 것을 발견했어. 스킬에서 필터링을 언급하고 있지만, 충분히 눈에 띄지 않는 것 같아?"



4. Claude A의 제안을 검토합니다: Claude A는 규칙을 더 눈에 띄게 재구성하거나, "always filter" 대신 "MUST filter"와 같은 더 강한 표현을 사용하거나, 워크플로우 섹션을 재구조화하는 것을 제안할 수 있습니다.



5. 변경을 적용하고 테스트합니다: Claude A의 개선 사항으로 스킬을 업데이트한 다음, 유사한 요청으로 Claude B에서 다시 테스트합니다.



6. 사용에 따라 반복합니다: 새로운 시나리오를 접할 때마다 이 관찰-개선-테스트 사이클을 계속합니다. 각 반복은 가정이 아닌 실제 에이전트 동작에 기반하여 스킬을 개선합니다.



팀 피드백 수집:

1. 팀원들과 스킬을 공유하고 사용 현황을 관찰합니다.

2. 질문합니다: 스킬이 예상대로 활성화되는가? 지시사항이 명확한가? 무엇이 빠져 있는가?

3. 피드백을 반영하여 자신의 사용 패턴에서 발견하지 못한 맹점을 해결합니다.



이 접근 방식이 효과적인 이유: Claude A는 에이전트의 필요를 이해하고, 사용자는 도메인 전문 지식을 제공하고, Claude B는 실제 사용을 통해 격차를 드러내며, 반복적 개선은 가정이 아닌 관찰된 동작에 기반하여 스킬을 개선합니다.



Claude가 스킬을 탐색하는 방식을 관찰하라


스킬을 반복 개선하면서 Claude가 실제로 어떻게 스킬을 사용하는지 주목하세요. 다음을 살펴보세요:



예상치 못한 탐색 경로: Claude가 예상하지 못한 순서로 파일을 읽나요? 이는 구조가 생각만큼 직관적이지 않을 수 있음을 나타냅니다.



놓친 연결: Claude가 중요한 파일에 대한 참조를 따르지 못하나요? 링크를 더 명시적이거나 눈에 띄게 만들어야 할 수 있습니다.



특정 섹션에 대한 과도한 의존: Claude가 같은 파일을 반복적으로 읽는다면, 해당 콘텐츠를 메인 SKILL.md에 넣는 것을 고려하세요.



무시된 콘텐츠: Claude가 번들된 파일에 접근하지 않는다면, 해당 파일이 불필요하거나 메인 지시사항에서 제대로 알려지지 않은 것일 수 있습니다.



가정이 아닌 이러한 관찰에 기반하여 반복하세요. 스킬 메타데이터의 name과 description은 특히 중요합니다. Claude는 현재 작업에 대한 응답으로 스킬을 트리거할지 결정할 때 이것을 사용합니다. 스킬이 무엇을 하고 언제 사용해야 하는지 명확하게 설명되어 있는지 확인하세요.

Windows 스타일 경로를 피하라


Windows에서도 파일 경로에 항상 슬래시(/)를 사용하세요:



✓ 좋음: scripts/helper.py, reference/guide.md

Unix 스타일 경로는 모든 플랫폼에서 작동하지만, Windows 스타일 경로는 Unix 시스템에서 오류를 발생시킵니다.



너무 많은 옵션을 제시하지 마라


필요하지 않다면 여러 접근 방식을 제시하지 마세요:



원문:

**Bad example: Too many choices** (confusing):
"You can use pypdf, or pdfplumber, or PyMuPDF, or pdf2image, or..."

**Good example: Provide a default** (with escape hatch):
"Use pdfplumber for text extraction:
```python
import pdfplumber
```

For scanned PDFs requiring OCR, use pdf2image with pytesseract instead."
번역:

나쁜 예시: 너무 많은 선택지 (혼란스러움): "pypdf, pdfplumber, PyMuPDF, pdf2image 중 아무거나 쓸 수 있어..."

좋은 예시: 기본값 제공 (탈출구 포함): "텍스트 추출에는 pdfplumber를 사용하세요. OCR이 필요한 스캔된 PDF에는 대신 pdf2image와 pytesseract를 사용하세요."



고급: 실행 가능한 코드가 포함된 스킬


아래 섹션들은 실행 가능한 스크립트를 포함하는 스킬에 초점을 맞춥니다. 스킬이 마크다운 지시사항만 사용한다면 효과적인 스킬을 위한 체크리스트로 건너뛰세요.



해결하라, 떠넘기지 마라


스킬용 스크립트를 작성할 때 오류 조건을 처리하세요. Claude에게 떠넘기지 마세요.



좋은 예시: 오류를 명시적으로 처리:

def process_file(path):
    """Process a file, creating it if it doesn't exist."""
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        # Create file with default content instead of failing
        print(f"File {path} not found, creating default")
        with open(path, "w") as f:
            f.write("")
        return ""
    except PermissionError:
        # Provide alternative instead of failing
        print(f"Cannot access {path}, using default")
        return ""


나쁜 예시: Claude에게 떠넘기기:

def process_file(path):
    # Just fail and let Claude figure it out
    return open(path).read()


설정 파라미터도 정당화되고 문서화되어야 합니다. "부두 상수(Voodoo Constants)"(Ousterhout의 법칙)를 피하세요. 올바른 값을 모르면 Claude가 어떻게 결정하겠습니까?



좋은 예시: 자기 설명적:

# HTTP requests typically complete within 30 seconds
# Longer timeout accounts for slow connections
REQUEST_TIMEOUT = 30

# Three retries balances reliability vs speed
# Most intermittent failures resolve by the second retry
MAX_RETRIES = 3


나쁜 예시: 매직 넘버:

TIMEOUT = 47  # Why 47?
RETRIES = 5  # Why 5?


유틸리티 스크립트를 제공하라


Claude가 스크립트를 작성할 수 있더라도, 미리 만들어진 스크립트에는 장점이 있습니다:



유틸리티 스크립트의 이점: 생성된 코드보다 더 안정적 / 토큰을 절약(컨텍스트에 코드를 포함할 필요 없음) / 시간을 절약(코드 생성 불필요) / 사용 간 일관성 보장



지시사항에서 Claude가 다음 중 어떤 것을 해야 하는지 명확하게 하세요:

스크립트 실행 (가장 일반적): "Run analyze_form.py to extract fields"

참조로 읽기 (복잡한 로직용): "See analyze_form.py for the field extraction algorithm"



대부분의 유틸리티 스크립트에서는 실행이 더 안정적이고 효율적이므로 선호됩니다.



예시:

## Utility scripts

**analyze_form.py**: Extract all form fields from PDF

```bash
python scripts/analyze_form.py input.pdf > fields.json
```

Output format:
```json
{
  "field_name": {"type": "text", "x": 100, "y": 200},
  "signature": {"type": "sig", "x": 150, "y": 500}
}
```

**validate_boxes.py**: Check for overlapping bounding boxes

```bash
python scripts/validate_boxes.py fields.json
# Returns: "OK" or lists conflicts
```

**fill_form.py**: Apply field values to PDF

```bash
python scripts/fill_form.py input.pdf fields.json output.pdf
```


시각적 분석을 활용하라


입력을 이미지로 렌더링할 수 있는 경우, Claude에게 분석시키세요:



## Form layout analysis

1. Convert PDF to images:
   ```bash
   python scripts/pdf_to_images.py form.pdf
   ```

2. Analyze each page image to identify form fields
3. Claude can see field locations and types visually


이 예시에서는 pdf_to_images.py 스크립트를 직접 작성해야 합니다. Claude의 비전(Vision) 기능은 레이아웃과 구조를 이해하는 데 도움이 됩니다.



검증 가능한 중간 산출물을 만들어라


Claude가 복잡하고 개방적인 작업을 수행할 때 실수를 할 수 있습니다. "계획-검증-실행(Plan-Validate-Execute)" 패턴은 Claude가 먼저 구조화된 형식으로 계획을 만들고, 실행 전에 스크립트로 해당 계획을 검증하게 하여 오류를 조기에 잡아냅니다.



예시: 스프레드시트를 기반으로 PDF의 50개 양식 필드를 업데이트하도록 Claude에게 요청한다고 상상하세요. 검증 없이는 Claude가 존재하지 않는 필드를 참조하거나, 충돌하는 값을 만들거나, 필수 필드를 놓치거나, 업데이트를 잘못 적용할 수 있습니다.



해결책: 위에서 보여준 워크플로우 패턴(PDF 양식 채우기)을 사용하되, 변경 사항을 적용하기 전에 검증되는 중간 changes.json 파일을 추가합니다. 워크플로우는 분석 → 계획 파일 생성 → 계획 검증 → 실행 → 확인이 됩니다.



이 패턴이 효과적인 이유:

오류 조기 포착: 검증이 변경 적용 전에 문제를 찾습니다.

기계 검증 가능: 스크립트가 객관적인 검증을 제공합니다.

되돌릴 수 있는 계획: Claude가 원본을 건드리지 않고 계획을 반복할 수 있습니다.

명확한 디버깅: 오류 메시지가 특정 문제를 가리킵니다.



사용 시점: 배치 작업, 파괴적 변경, 복잡한 검증 규칙, 고위험 작업.



구현 팁:

원문:

"Field 'signature_date' not found. Available fields: customer_name, order_total, signature_date_signed"

번역:

검증 스크립트를 구체적인 오류 메시지로 상세하게 만드세요. 예: "필드 'signature_date'를 찾을 수 없습니다. 사용 가능한 필드: customer_name, order_total, signature_date_signed" — 이렇게 하면 Claude가 문제를 수정하는 데 도움이 됩니다.



패키지 의존성


스킬은 플랫폼별 제한이 있는 코드 실행 환경에서 실행됩니다:



claude.ai: npm과 PyPI에서 패키지를 설치하고 GitHub 저장소에서 가져올 수 있습니다.

Claude API: 네트워크 접근이 없으며 런타임 패키지 설치가 불가능합니다.



필요한 패키지를 SKILL.md에 나열하고 코드 실행 도구 문서에서 사용 가능한지 확인하세요.



런타임 환경


스킬은 파일 시스템 접근, bash 명령, 코드 실행 기능이 있는 코드 실행 환경에서 실행됩니다. 이 아키텍처의 개념적 설명은 개요의 스킬 아키텍처를 참고하세요.



작성에 영향을 미치는 사항:



Claude가 스킬에 접근하는 방식:

1. 메타데이터 사전 로드: 시작 시 모든 스킬의 YAML 프론트매터에서 이름과 설명이 시스템 프롬프트에 로드됩니다.

2. 파일 온디맨드 읽기: Claude는 필요할 때 bash Read 도구를 사용하여 파일 시스템에서 SKILL.md와 기타 파일에 접근합니다.

3. 효율적인 스크립트 실행: 유틸리티 스크립트는 전체 내용을 컨텍스트에 로드하지 않고 bash를 통해 실행할 수 있습니다. 스크립트의 출력만 토큰을 소비합니다.

4. 대용량 파일의 컨텍스트 페널티 없음: 참조 파일, 데이터, 문서는 실제로 읽힐 때까지 컨텍스트 토큰을 소비하지 않습니다.



파일 경로가 중요합니다: Claude는 파일 시스템처럼 스킬 디렉터리를 탐색합니다. 백슬래시가 아닌 슬래시(reference/guide.md)를 사용하세요.

파일 이름을 설명적으로 지정하세요: doc2.md가 아닌 form_validation_rules.md처럼 내용을 나타내는 이름을 사용하세요.

발견을 위해 구성하세요: 도메인이나 기능별로 디렉터리를 구조화하세요. 좋음: reference/finance.md, reference/sales.md. 나쁨: docs/file1.md, docs/file2.md

포괄적인 리소스를 번들하세요: 완전한 API 문서, 광범위한 예시, 대규모 데이터셋을 포함하세요. 접근할 때까지 컨텍스트 페널티가 없습니다.

결정적 작업에는 스크립트를 선호하세요: Claude에게 검증 코드를 생성하도록 요청하기보다 validate_form.py를 작성하세요.

실행 의도를 명확히 하세요: "Run analyze_form.py to extract fields" (실행) vs "See analyze_form.py for the extraction algorithm" (참조로 읽기)

파일 접근 패턴을 테스트하세요: 실제 요청으로 테스트하여 Claude가 디렉터리 구조를 탐색할 수 있는지 확인하세요.



예시:

bigquery-skill/
├── SKILL.md (개요, 참조 파일을 가리킴)
└── reference/
    ├── finance.md (매출 지표)
    ├── sales.md (파이프라인 데이터)
    └── product.md (사용 분석)


사용자가 매출에 대해 물으면 Claude는 SKILL.md를 읽고 reference/finance.md에 대한 참조를 확인한 후 bash를 호출하여 해당 파일만 읽습니다. sales.md와 product.md 파일은 파일 시스템에 남아 있으며 필요할 때까지 컨텍스트 토큰을 전혀 소비하지 않습니다. 이 파일 시스템 기반 모델이 점진적 공개를 가능하게 합니다. Claude는 각 작업에 필요한 것만 정확히 탐색하고 선택적으로 로드할 수 있습니다.



기술 아키텍처에 대한 자세한 내용은 스킬 개요의 스킬 작동 방식을 참고하세요.



MCP 도구 참조


스킬이 MCP(Model Context Protocol) 도구를 사용하는 경우, "도구를 찾을 수 없음" 오류를 피하기 위해 항상 정규화된 도구 이름(Fully Qualified Tool Name)을 사용하세요.



형식: ServerName:tool_name



원문:

Use the BigQuery:bigquery_schema tool to retrieve table schemas.
Use the GitHub:create_issue tool to create issues.
번역:

테이블 스키마를 검색하려면 BigQuery:bigquery_schema 도구를 사용하세요.
이슈를 생성하려면 GitHub:create_issue 도구를 사용하세요.


여기서 BigQuery와 GitHub는 MCP 서버 이름이고, bigquery_schema와 create_issue는 해당 서버 내의 도구 이름입니다. 서버 접두어 없이는 특히 여러 MCP 서버가 사용 가능할 때 Claude가 도구를 찾지 못할 수 있습니다.



도구가 설치되어 있다고 가정하지 마라


패키지가 사용 가능하다고 가정하지 마세요:



원문:

**Bad example: Assumes installation**:
"Use the pdf library to process the file."

**Good example: Explicit about dependencies**:
"Install required package: `pip install pypdf`

Then use it:
```python
from pypdf import PdfReader
reader = PdfReader("file.pdf")
```"
번역:

나쁜 예시: 설치를 가정함: "pdf 라이브러리를 사용하여 파일을 처리하세요."

좋은 예시: 의존성을 명시함: "필요한 패키지를 설치하세요: pip install pypdf. 그 다음 사용하세요."

효과적인 스킬을 위한 체크리스트


스킬을 공유하기 전에 확인하세요:



핵심 품질

☐ 설명이 구체적이고 핵심 용어를 포함한다

☐ 설명에 스킬이 무엇을 하고 언제 사용해야 하는지 모두 포함되어 있다

☐ SKILL.md 본문이 500줄 이하이다

☐ 추가 세부 사항은 별도 파일에 있다 (필요한 경우)

☐ 시간에 민감한 정보가 없다 (또는 "이전 패턴" 섹션에 있다)

☐ 전체적으로 일관된 용어를 사용한다

☐ 예시가 추상적이지 않고 구체적이다

☐ 파일 참조가 한 단계 깊이이다

☐ 점진적 공개가 적절히 사용되었다

☐ 워크플로우에 명확한 단계가 있다



코드와 스크립트

☐ 스크립트가 Claude에게 떠넘기지 않고 문제를 해결한다

☐ 오류 처리가 명시적이고 도움이 된다

☐ "부두 상수"가 없다 (모든 값이 정당화됨)

☐ 필요한 패키지가 지시사항에 나열되어 있고 사용 가능한지 확인되었다

☐ 스크립트에 명확한 문서가 있다

☐ Windows 스타일 경로가 없다 (모두 슬래시)

☐ 중요한 작업에 대한 검증/확인 단계가 있다

☐ 품질이 중요한 작업에 피드백 루프가 포함되어 있다



테스트

☐ 최소 세 개의 평가가 생성되었다

☐ Haiku, Sonnet, Opus에서 테스트되었다

☐ 실제 사용 시나리오로 테스트되었다

☐ 팀 피드백이 반영되었다 (해당되는 경우)
